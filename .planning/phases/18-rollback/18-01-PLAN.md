**bot/rollback.py (new file):**

Create rollback execution logic modeled closely on bot/deploy.py. The module handles both self-rollback (super_bot) and external rollback (mic_transformer).

`handle_rollback(repo_name, repo_config, client, channel, thread_ts, user_id, ack_ts=None, target_sha=None)`:

Step 1 - Determine rollback target SHA:
- If `target_sha` is provided by the user, use it directly.
- If not, call `get_last_deploy(repo_name)` and use the `pre_sha` field as the rollback target.
- If no deploy history or no `pre_sha` in history, edit ack message with: "No previous deploy found for {repo_name}. Specify a SHA: `rollback {repo_name} abc1234`" and return.

Step 2 - Validate target SHA exists:
- Run `git cat-file -t {target_sha}` in the repo dir to verify the SHA exists. If it fails, edit ack message with "SHA `{target_sha}` not found in {repo_name} repository" and return.

Step 3 - Get current SHA (this is the pre-rollback state for auto-roll-forward):
- Get current HEAD short SHA via `_git(repo_dir, "rev-parse", "--short", "HEAD")`.

Step 4 - Show rollback info (informational, no gate):
- Show what commits will be rolled back: run `git log --oneline {target_sha}..HEAD` to show what's being undone.
- Edit ack message: "Rolling back {repo_name}...\n`{current_sha}` -> `{target_sha}`\n```\n{commits_being_undone}\n```"

Step 5 - Branch: self-rollback vs external rollback.

For self-rollback (`repo_config.get("self_deploy")`):
- Call `write_deploy_state(channel, thread_ts, current_sha, user_id, action="rollback")` -- current_sha is stored so roll-forward knows where to go.
- Edit ack message to add "Restarting now for rollback. I'll be back shortly."
- Trigger Prefect deploy with `{"branch": target_sha}` (the deploy script does `git checkout {branch}` so passing the SHA works).
- The bot process dies; recovery happens in app.py on restart.

For external rollback (mic_transformer):
- Trigger Prefect deploy with `{"branch": target_sha}`.
- Poll Prefect status (reuse same poll pattern as deploy.py: POLL_INTERVAL=5, MAX_POLL_DURATION=600, check TERMINAL_STATES).
- On COMPLETED: run health check via `_health_check(repo_name, repo_config)`.
  - If health check passes: call `record_deploy(repo_name, target_sha[:7], pre_sha=current_sha)`, edit message "Rollback {repo_name} complete. Now on `{target_sha[:7]}`."
  - If health check fails: **auto-roll-forward** -- trigger another Prefect deploy with `{"branch": current_sha}`, poll for completion.
    - If roll-forward succeeds AND health check passes: edit message "Rollback failed health check. Automatically rolled forward to `{current_sha}`. Reason: {health_failure_reason}"
    - If roll-forward fails or health check fails: edit message "Rollback failed and auto-roll-forward also failed. Manual SSH intervention needed. Last known state: {details}"
- On FAILED/CANCELLED/CRASHED: same auto-roll-forward logic.

`_health_check(repo_name, repo_config)` -> tuple[bool, str]:
- For repos with a `service` key: run `systemctl is-active {service}` and check last 10 journal lines for "error" or "traceback" (case-insensitive). Return (True, "") if service is active and no errors, else (False, reason).
- For repos without a `service` key (mic_transformer currently has `service: None`): just verify the Prefect deploy completed successfully (the COMPLETED state is sufficient). Return (True, "").

Import `_git` from `bot.deploy_state` (already exported). Import `prefect_api` and `deploy_state` functions as needed. Use `_edit_progress` helper (copy from deploy.py or import -- simplest to define locally since it's 4 lines).

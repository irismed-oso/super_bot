Create `bot/deploy_state.py` with these components:

1. **REPO_CONFIG dict** mapping canonical repo names to their config:
   ```python
   REPO_CONFIG = {
       "super_bot": {
           "dir": "/home/bot/super_bot",
           "prefect_deployment": "deploy-superbot",
           "service": "superbot",
           "aliases": ["superbot", "super_bot", "sb"],
           "self_deploy": True,  # requires restart handling
       },

           "self_deploy": False,
       },
   }
   ```

2. **resolve_repo(text: str) -> tuple[str, dict] | None** -- resolve a repo name or alias from text to (canonical_name, config_dict). Check all aliases for all repos. Return None if no match.

3. **Deploy-state file I/O** (for self-deploy recovery):
   - `DEPLOY_STATE_PATH = "/home/bot/.deploy-state.json"`
   - `write_deploy_state(channel, thread_ts, pre_sha, user_id) -> None` -- write JSON with those fields plus `triggered_at` timestamp
   - `read_and_clear_deploy_state() -> dict | None` -- read and delete the file. Return None if not found or if older than 5 minutes (stale check using `triggered_at`). Catch JSONDecodeError and OSError gracefully.

4. **Async git helpers** for status and preview:
   - `async def _git(repo_dir, *args) -> str` -- run git command in repo dir via asyncio.create_subprocess_exec, return stdout stripped. Raise RuntimeError on non-zero exit.
   - `async def get_repo_status(repo_name: str) -> dict` -- returns `{"sha": str, "branch": str, "behind": int, "dirty": bool}`. Runs: `git rev-parse --short HEAD`, `git rev-parse --abbrev-ref HEAD`, `git fetch origin main --quiet`, `git rev-list --count HEAD..origin/main`, and `git status --porcelain` (dirty if output non-empty).
   - `async def get_deploy_preview(repo_name: str) -> str` -- runs `git fetch origin main --quiet` then `git log --oneline HEAD..origin/main` and returns the output (or "Already on latest -- nothing to deploy." if empty).

Use structlog for logging. No new dependencies.

<task type="auto">
  <name>Task 2: Add deploy status, preview, and active-task guard to fast_commands.py</name>
  <files>bot/fast_commands.py</files>
  <action>
Add three new fast-path handlers to `bot/fast_commands.py`:

1. **Deploy status handler** (`_handle_deploy_status`):
   - Regex: `_DEPLOY_STATUS_RE = re.compile(r"deploy\s+(?:status|info)(?:\s+(\S+))?", re.IGNORECASE)`
   - If a repo name is provided, show status for just that repo. Otherwise show status for all repos in REPO_CONFIG.
   - Call `get_repo_status()` from deploy_state for each repo.
   - Format output as:
     ```
     *super_bot* (`main` @ `abc1234`)
       3 commits behind origin/main

     *mic_transformer* (`main` @ `def5678`)
       Up to date
     ```
   - If repo has dirty state, append " (uncommitted changes)" to the line.

2. **Deploy preview handler** (`_handle_deploy_preview`):
   - Regex: `_DEPLOY_PREVIEW_RE = re.compile(r"deploy\s+preview\s+(\S+)", re.IGNORECASE)`
   - Requires a repo name (error if missing).
   - Call `get_deploy_preview()` from deploy_state.
   - Return the commit list or "nothing to deploy" message.

3. **Deploy guard handler** (`_handle_deploy_guard`):
   - Regex: `_DEPLOY_GUARD_RE = re.compile(r"deploy\s+(?:force\s+)?(\S+)\s*$", re.IGNORECASE)` -- matches "deploy superbot", "deploy force superbot"
   - This is NOT a full deploy handler -- it only checks whether to block/warn.
   - If `queue_manager.get_current_task()` is not None AND "force" is NOT in the text:
     - Return a warning: "An agent task is currently running. Use `deploy force [repo]` to proceed anyway."
   - Otherwise return None (fall through to agent pipeline for actual deploy).
   - Per CONTEXT.md: deploy commands go through the agent pipeline. This guard only provides the warning before the message hits the queue.

**Register in FAST_COMMANDS list** -- order matters:
- Deploy status and preview BEFORE the deploy guard (so "deploy status" and "deploy preview" don't fall through to the guard).
- Deploy guard at the end of the deploy-related entries.
- All deploy entries BEFORE the existing bot_status entry but AFTER the crawl/status entries.

Import from deploy_state: `from bot.deploy_state import get_repo_status, get_deploy_preview, resolve_repo, REPO_CONFIG`

**Important:** The deploy guard handler must return None (not a string) when the deploy should proceed, so the message falls through to the agent pipeline. The guard is a "block or pass" check, not a deploy executor.

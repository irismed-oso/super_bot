Create `bot/deploy.py` with the deploy execution logic. This module handles both super_bot (self-deploy with restart) and mic_transformer (deploy with polling).

1. **`async def handle_deploy(repo_name: str, repo_config: dict, client, channel: str, thread_ts: str, user_id: str, ack_ts: str | None = None) -> None`**

   This is called from the agent pipeline (handlers.py) after the deploy guard passes. It edits the ack message in-place as each step completes.

   **Common flow (both repos):**
   a. Check if already on latest: call `get_repo_status()`. If `behind == 0`, edit ack message to "Already on latest (`{sha}`). Nothing to deploy." and return.
   b. If dirty state: include a warning line "(uncommitted changes on VM -- proceeding anyway)" per CONTEXT.md.
   c. Get deploy preview: call `get_deploy_preview()` to show what will be deployed.
   d. Find Prefect deployment: call `prefect_api.find_deployment_id(repo_config["prefect_deployment"])`. If not found, edit ack with error and return.
   e. Trigger Prefect flow run: call `prefect_api.create_flow_run(deployment_id, {"branch": "main"})`. Always deploys main per CONTEXT.md.

   **super_bot self-deploy flow (repo_config["self_deploy"] is True):**
   f. BEFORE triggering Prefect (after finding deployment): write deploy-state file via `write_deploy_state(channel, thread_ts, pre_sha, user_id)`.
   g. Edit ack message with detailed pre-restart message per CONTEXT.md:
      ```
      Deploying super_bot...
      `{pre_sha}` -> `origin/main`
      {preview_commits}

      Restarting now. I'll be back shortly.
      If I don't reply within 30 seconds, check logs: `sudo journalctl -u superbot -n 50`
      ```
   h. Trigger the Prefect flow run (this will restart the bot).
   i. No polling -- the bot process will die. Post-restart recovery handles the "I'm back" message.

   **mic_transformer deploy flow (self_deploy is False):**
   f. Edit ack message: "Deploying mic_transformer... triggered Prefect flow `{run_name}`"
   g. Poll `prefect_api.get_flow_run_status(flow_run_id)` every 5 seconds until terminal state.
   h. Edit ack message with each status change (like heartbeat edits).
   i. On COMPLETED: edit ack with success message including new commit SHA (run `git rev-parse --short HEAD` in mic_transformer dir after deploy).
   j. On FAILED/CRASHED: edit ack with failure message and Prefect state message.

   Use `client.chat_update(channel=channel, ts=ack_ts, text=...)` for edit-in-place updates.

2. **Helper: `async def _edit_progress(client, channel, ts, text)`** -- wrapper around `client.chat_update` with error handling.

3. **Constants:**
   - `POLL_INTERVAL = 5` (seconds between Prefect status checks)
   - `MAX_POLL_DURATION = 600` (10 min max poll time for mic_transformer)
   - `TERMINAL_STATES = frozenset({"COMPLETED", "FAILED", "CANCELLED", "CRASHED"})`

Use structlog throughout. Import from `bot.deploy_state` (Plan 01) and `bot.prefect_api`.

**bot/app.py changes:**

Add a deploy-state recovery check that runs AFTER the Slack Socket Mode connection is established. Per CONTEXT.md: "Post-restart confirmation fires as soon as Slack connection is re-established."

1. Add `_check_deploy_recovery(client)` async function:
   - Call `read_and_clear_deploy_state()` from `bot.deploy_state`.
   - If None, return (no pending deploy).
   - Get current commit SHA via `asyncio.create_subprocess_exec("git", "rev-parse", "--short", "HEAD", cwd="/home/bot/super_bot")`.
   - Post to the original thread: `"I'm back, running commit \`{current_sha}\`."` using `client.chat_postMessage(channel=state["channel"], thread_ts=state["thread_ts"], text=...)`.
   - Log the recovery event.

2. In `main()`, schedule the recovery check AFTER `handler.start_async()` is called. Since `start_async()` blocks (it's the main loop), use `asyncio.create_task()` BEFORE `start_async()` with a brief delay to ensure the WebSocket connection is established:
   ```python
   async def _delayed_deploy_check():
       await asyncio.sleep(5)  # wait for Socket Mode connection
       await _check_deploy_recovery(app.client)

   asyncio.create_task(_delayed_deploy_check())
   ```
   Place this just before `await handler.start_async()`.

**bot/handlers.py changes:**

Add deploy command detection and routing in the `_run_agent_real` function. Deploy commands go through the agent pipeline per CONTEXT.md, but we need to intercept them to call `handle_deploy` instead of running a full Claude agent session.

1. Add a deploy detection regex at module level:
   ```python
   _DEPLOY_CMD_RE = re.compile(
       r"deploy\s+(?:force\s+)?(\S+)\s*$",
       re.IGNORECASE,
   )
   ```

2. In `_run_agent_real`, AFTER the fast-command check but BEFORE enqueueing to the agent queue, check if the message matches `_DEPLOY_CMD_RE`:
   - If matched, resolve the repo via `deploy_state.resolve_repo(text)`.
   - If repo not resolved, post "Unknown repo. Available: super_bot, mic_transformer" to thread and return.
   - Call `handle_deploy(repo_name, repo_config, client, channel, thread_ts, user_id, ack_ts=ack_ts)` directly (not through the queue).
   - Return after deploy completes (do not fall through to agent queue).

3. Import: `from bot.deploy_state import resolve_repo` and `from bot.deploy import handle_deploy`.

**Important:** The deploy command is handled outside the agent queue because:
- For super_bot: the bot will die during deploy, so it can't be queued
- For mic_transformer: the polling loop runs in the handler directly
- The active-task guard in fast_commands.py already blocks deploys when a task is running (unless "force" is used)

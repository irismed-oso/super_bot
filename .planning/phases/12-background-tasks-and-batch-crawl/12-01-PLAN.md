<task type="auto">
  <name>Task 2: Background monitor with progress polling and final summary</name>
  <files>bot/background_monitor.py</files>
  <action>
Create `bot/background_monitor.py` with the background polling system:

1. **`start_batch_monitor(slack_context: dict, runs: list[tuple], date_str: str) -> None`**
   - `slack_context` has `client`, `channel`, `thread_ts`
   - `runs` is list of `(location_name, flow_run_id, flow_run_name)` tuples
   - Wraps `_monitor_loop` in `asyncio.create_task()` so it runs independently of the caller
   - Logs the task creation for observability

2. **`_monitor_loop(slack_context, runs, date_str) -> None`** (async, private):
   - Maintains a dict mapping `flow_run_id -> {"location": str, "name": str, "status": str, "message": str | None}`
   - Initialize all statuses to "PENDING"
   - Poll loop with `POLL_INTERVAL = 30` seconds (polls every 30s, but only posts Slack updates every `UPDATE_INTERVAL = 150` seconds / 2.5 minutes)
   - Each poll iteration: call `prefect_api.get_flow_run_status(run_id)` for all non-terminal runs using `asyncio.gather(return_exceptions=True)`. Terminal states: COMPLETED, FAILED, CANCELLED, CRASHED.
   - Update the status dict with new states
   - Every UPDATE_INTERVAL seconds (tracked by a `last_update_time` timestamp), post a progress message to the Slack thread via `client.chat_postMessage()`:
     ```
     Batch crawl progress ({date_str}):
     Completed: {N} | Running: {N} | Pending: {N} | Failed: {N}

     Recently finished:
     - {location}: completed
     - {location}: failed - {error message}
     ```
   - When ALL runs are terminal (no RUNNING/PENDING/SCHEDULED left), post the final summary and exit the loop
   - Wrap entire loop in try/except to log errors -- background task must not crash silently

3. **`_format_progress(statuses: dict, date_str: str) -> str`** -- Format intermediate progress update.
   Group by state: completed, running (RUNNING/PENDING/SCHEDULED), failed (FAILED/CANCELLED/CRASHED).
   Show counts and list recently finished locations.

4. **`_format_final_summary(statuses: dict, date_str: str) -> str`** -- Format the final summary.
   Group all locations by outcome:
   - Completed: list locations
   - Failed: list locations with error messages from `state.message`
   - Format as a clear Slack message:
     ```
     Batch crawl complete ({date_str}):
     {N} succeeded | {N} failed

     Completed ({N}):
     {bullet list of locations}

     Failed ({N}):
     {bullet list with error messages}
     ```
   If all succeeded: omit the Failed section. If none failed: just show "All {N} locations completed successfully."

5. **Safety:**
   - `MAX_POLL_DURATION = 3600` seconds (1 hour). If the loop runs longer than this, post a timeout message and exit. EyeMed crawls typically take 5-20 minutes per location.
   - Handle httpx errors gracefully in polling -- if a single status check fails, log it but continue polling (don't abort the whole monitor)
   - Use `structlog` for all logging with structured fields (run_count, completed_count, etc.)

**Critical design point:** This runs as an `asyncio.create_task()` in the same event loop as the bot. It does NOT touch the agent queue at all -- it's just async HTTP calls + Slack API calls. This is what satisfies BGTK-03 (no queue blocking).
